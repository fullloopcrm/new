## Fresh angles checked this pass, all clean

- **CORS wildcards**: only 2 routes set `Access-Control-Allow-Origin: '*'`
  (`track`, `leads/visits`) — both are write-only public analytics beacons
  (tracking pixel / visit logging), no `Access-Control-Allow-Credentials`,
  no cookie-based auth, no sensitive data returned. Standard pattern for
  cross-origin tracking endpoints, not a gap.
- **Open redirect via OAuth callbacks** (`google/callback`,
  `admin/google/callback`, `social/connect/facebook/callback`,
  `social/connect/instagram/callback`): all redirect targets are
  `${baseUrl}/<fixed-path>?...`, `baseUrl` sourced from
  `NEXT_PUBLIC_APP_URL` env var (not Host header, not a caller-supplied
  param) — no user-controlled redirect target anywhere in this set.
- **Email-subject CRLF/header injection**: `sendEmail()`
  (`src/lib/email.ts`) goes through the Resend SDK's JSON REST API
  (`client.emails.send()`), not raw SMTP text concatenation — a newline in
  an interpolated subject (`client.name`, `booking.clientName`, etc.) stays
  a literal character in a JSON field, no header-injection vector the way
  classic PHP `mail()`/raw-SMTP CRLF injection works.
- **CSV/ZIP export endpoints** (`clients/[id]/export`,
  `cpa/[token]/year-end-zip`, `finance/tax-export`,
  `finance/year-end-zip`, `admin/businesses/[id]/site-export`): all
  tenant-scoped; `clients/[id]/export` already neutralizes CSV
  formula-injection (`=`,`+`,`-`,`@`,tab,CR prefix quoting) and sanitizes
  filenames going into `Content-Disposition`. `cpa/[token]` tokens are
  `randomBytes(24)` (192-bit) with expiry + revocation checks — brute-force
  infeasible regardless of rate limiting.
- **Stripe webhook** (`webhooks/stripe/route.ts`): fails closed if
  `STRIPE_WEBHOOK_SECRET` is unset (500, never proceeds to
  `constructEvent`), signature verified via SDK before any event handling —
  same fail-closed class already fixed for `CRON_SECRET`/Clerk/Resend in
  prior sessions.

## One observation — not fixed, flagging for the leader

`error.message` from caught exceptions/Supabase errors is returned
verbatim to the client in **368 call sites** across `src/app/api`
(`NextResponse.json({ error: error.message }, { status: 500 })` and
equivalents). Most are Postgres/PostgREST error strings (constraint names,
column types) rather than stack traces — moderate-severity info disclosure
at worst, not a "clearly live-exploitable" bug. Not attempting a fix this
pass: touching 368 sites to introduce generic client-facing messages while
preserving server-side detail (via a logger) is a large, cross-cutting
change with real regression risk (some frontend code may pattern-match on
specific error strings), the opposite of the "lower-risk, narrow" fix this
order asks for. Recommend as a dedicated pass (introduce a
`safeErrorResponse()` helper, roll out incrementally) if the leader wants
it — not a drive-by sweep.
