## 2. New fix: `portal/connect` POST channel_id IDOR

`deploy-prep/supabase-admin-gate-baseline-audit.md` (earlier W4 session)
flagged this as a real-but-unexploited gap and explicitly left it unfixed
("recommend someone add the ownership check as routine hardening"). Picked
it up this pass since it's exactly the class of low-risk, well-scoped fix
the order is asking for.

**The gap:** `POST /api/portal/connect` accepts a caller-supplied
`channel_id` in the request body and uses it as-is to insert into
`connect_messages` — no check that the authenticated portal client actually
owns that channel. `tenantDb().insert()` stamps `tenant_id`, so this can't
cross a tenant boundary, but within the tenant a client who obtains another
party's channel UUID (log exposure, a future unrelated leak, a compromised
staff session) could inject a message into it. The GET handler and the
"no `channel_id` supplied" branch of POST both already resolve the channel
via `.eq('type','client').eq('client_id', auth.id)` — only the
caller-supplied-`channel_id` branch skipped that check.

**Fix:** when `channel_id` is present in the POST body, verify it resolves
to a channel owned by the authenticated client
(`tenantDb(auth.tid).from('connect_channels').eq('id', targetChannelId).eq('type','client').eq('client_id', auth.id).single()`)
before using it; return 403 if not found. Matches the ownership-check
pattern already used elsewhere in this file and in `client/properties`.

**Compatibility check:** read the only caller (`src/app/portal/connect/page.tsx`)
— it always echoes back the `channel_id` it received from its own prior GET
response (i.e. its own resolved channel), so this is fully backward
compatible with the real frontend; only a forged/off-path request is
affected.
