# Jefe — Full Loop Platform GM: Wiring Plan

**Mandate (your words):** Jefe does *not* care about any tenant's revenue, clients, or operations. Jefe cares about **Full Loop itself** — is it working, is it growing — and about **Full Loop's leads & sales, security, stability, knowing tenant issues before they do, and communicating with tenants to fix issues immediately.**

Everything below is mapped to real tables/endpoints already in the repo (confirmed 2026-06-28).

---

## Pillar 1 — Growth (Full Loop's own sales)

Jefe owns the *product's* pipeline, not tenants' pipelines.

**Wire into:**
- `inquiries` table — contact-form leads from homeservicesbusinesscrm.com (currently 4)
- `prospects` table — outbound/qualified prospects
- `/api/admin/billing` — Full Loop's **own MRR**: active subscriptions, setup fees, churn
- `tenants.setup_progress` / wizard checklist — **onboarding stalls** (tenant stuck mid-setup = at-risk deal)

**Jefe should answer / alert:** new inquiries waiting, inquiries with no follow-up in X hrs, prospects going cold, onboarding stuck, MRR up/down, a tenant that churned.

## Pillar 2 — Security

**Wire into:**
- `security_events` table (auth failures, impersonation, cross-tenant access attempts)
- Admin posture panels (the security chat built these): checks, webhook status, **secret health**, impersonation audit
- `notifications` type `security` (382 logged)

**Jefe alerts:** spike in failed logins, any cross-tenant access attempt, impersonation events, exposed/expired secrets.

## Pillar 3 — Stability

**Wire into:**
- `notifications` types: `error` (898), `selena_error` (502), `comms_fail` (360), `comms_monitor_alert` (167), `schedule_issue` (147)
- `error_logs` table
- `/api/admin/system-check`, `/api/admin/selena/monitor`
- Vercel deploy status + per-tenant domain uptime (synthetic probe — I did this manually earlier)
- Cron health (reminders, payment, rating, comms-monitor) — did they fire?

**Jefe alerts:** error/selena_error/comms_fail spikes, a cron that didn't run, a deploy that failed, a tenant domain returning non-200.

## Pillar 4 — Proactive tenant-issue detection ("know before they do")

This is the differentiator. Per-tenant signal watch with thresholds → alert + recommend outreach:

| Signal (source) | Means | Jefe action |
|---|---|---|
| `comms_fail` / `comms_monitor_alert` spike | SMS/email breaking for that tenant | Flag + "reach out, their leads aren't getting replies" |
| Invalid Telnyx/Resend/Stripe key (validate like I did for Anthropic) | Integration down | Flag before tenant loses money |
| `selena_error` spike | Their agent is failing | Flag + investigate |
| `schedule_issue` | Scheduling broken | Flag |
| Sudden drop in a tenant's bookings/comms vs baseline | Something silently broke | Anomaly alert |
| Stripe webhook failing | Payments not reconciling | Flag |
| Onboarding stalled N days | Deal at risk | Flag for outreach |

## Pillar 5 — Communication & action

Jefe shouldn't just report — he reaches out and fixes:
- **Message a tenant owner** (SMS via their Telnyx / email via Resend / their Telegram) when an issue is detected
- Create a callback/task, log the outreach, track resolution
- Draft the message, you approve, Jefe sends (respects your no-mass-SMS / approval rules)
